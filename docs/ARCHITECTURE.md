# Architecture Overview

openCenter is designed to manage cluster configurations and GitOps scaffolding in a predictable, testable way. The implementation is intentionally simple: YAML files on disk represent the truth of a cluster; embedded templates produce GitOps manifests; and a small set of validation rules guard against configuration mistakes.

## Core Concepts

- **Configuration Store**: A directory on disk (default `~/.config/openCenter`) holds YAML files for each cluster. The filename without extension is the cluster name. A `.active` file stores the name of the currently selected cluster. The `internal/config` package encapsulates all logic around loading, saving, listing and validating these files.

- **CLI Commands**: The CLI is built with Cobra. The root command (`openCenter`) registers a `cluster` parent command with numerous subcommands (list, init, select, current, info, validate, preflight, setup, render, bootstrap, schema) and a separate `prompt` command. Each subcommand is implemented in its own Go file for clarity.

- **GitOps Templates**: Starter files for a GitOps repository live under `internal/gitops/templates`. These files are embedded at compile time using GoŌĆÖs `embed` directive. The `internal/gitops` package exposes a `CopyTemplates` function that either copies files verbatim or renders `.tmpl` files via GoŌĆÖs `text/template` engine with the [Sprig](https://masterminds.github.io/sprig/) function library loaded. Rendering binds the entire cluster configuration struct as the template context.

- **Validation and Schema**: Invariants across configuration fields are enforced by the `Validate` function. If any rule fails, the `validate` command prints messages and exits nonŌĆæzero. A simplified JSON schema matching the configuration shape is generated by `GenerateSchema` for IDE consumption.

- **Preflight Checks**: The `preflight` command detects the presence of required executables (`git`, `kubectl`, `talosctl`) and calls providerŌĆæspecific checks. For OpenStack, it warns about missing CLI tools or blank `auth_url`.

- **Prompt Integration**: The `prompt` command outputs a short string for inclusion in shell prompts. It is intentionally noŌĆæop outside a clusterŌĆÖs GitOps directory to avoid polluting unrelated shell sessions.

## Flow Examples

### Initialising a Cluster

1. The user runs `openCenter cluster init demo`.
2. The CLI calls `config.NewDefault("demo")` to obtain a struct with default values.
3. The struct is optionally updated via interactive prompts (if the name was not specified on the command line).
4. `config.Save` writes the YAML to `${config-dir}/demo.yaml` with secure file permissions.

### Setting Up GitOps

1. The user runs `openCenter cluster setup --render`.
2. The CLI loads the active clusterŌĆÖs configuration.
3. `gitops.CopyTemplates` iterates over the embedded files. For files with the `.tmpl` extension, it executes the template with the configuration; otherwise it copies the file directly. The destination path is `gitops.git_dir`.
4. A `.opencenter` marker file containing the cluster name is written.
5. If no `.git` directory exists, the CLI runs `git init -b main`.
6. The working tree is staged and committed using `git add .` and `git commit --allow-empty`.

### Bootstrapping

1. The user sets `gitops.git_url` in the YAML and runs `openCenter cluster bootstrap`.
2. The CLI stages and commits any outstanding changes.
3. It adds or updates the `origin` remote to the configured URL.
4. It pushes the `main` branch using `git push -u origin main`.
